# CITTAFRACTA Development Guidelines

## 1. Core Philosophy: The Historian's Code
* **Context is King:** Before starting any task, read `STATUS.md`. Do not start coding a feature if the previous refactor isn't stable.
* **Historical Accuracy > Fancy Code:** If a code solution works but distorts the historical data representation, it is rejected.
* **The AI Rule:** Never copy-paste AI code you don't understand. Ask "Why?" and "What does this break?" before implementing.

## 2. Workflow (Solo/Vibe Coding Edition)
* **Branching:** Use `feature/` branches for new ideas. Keep `main` distinct. Do not push broken code to `main`.
* **Commit Messages:** Write for your future self. Instead of "fix bug", write "Fix timeline slider not updating map year".
* **The "Sanity Check":** Before merging a feature, manually verify:
    1.  Does the Geo Map load?
    2.  Does the Social Map load?
    3.  Does the Timeline slider work?

## 3. Architecture & Data Integrity
* **Sheet Sanctity:** The Google Sheet structure is the source of truth. Any change to columns in the Sheet MUST be reflected immediately in `src/types.ts`.
* **Service Layer Protection:** Changes to `sheetService.ts` require extreme caution. This is the bridge between history and code.
* **Type Safety:** We use TypeScript constraints. Avoid `any` types. If the AI suggests `any`, ask for a strict type instead.

## 4. Security & Privacy
* **API Keys:** The Google Sheets API key must be in `.env.local` and never committed to GitHub.
* **Read-Only:** The application should only have READ access to the Google Sheet. Never write logic that attempts to modify the historical database from the client.

## 5. Maintenance
* **Boy Scout Rule:** If you see "lazy code" generated by AI (e.g., repeating logic instead of a reusable function), ask the AI to refactor it immediately.
* **Documentation:** Update `CONTEXT.md` if a major feature changes (e.g., if we add a new map layer).
